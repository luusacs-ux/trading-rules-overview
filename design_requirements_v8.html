2. Unified Rules Tab

2.1 Purpose

A single, searchable table showing every active trading rule. Replaces the need to look up rules per-ticker. Every rule on this list is active — there is no status column.

2.2 Rule ID Format

Naming Convention UPDATED from v6

Format: {TICKER}_{TYPE}{TF}_{SEQ5}_WR{wr}%/APnL{apnl}%/PTPnL{ptpnl}%

Example: AMZN_S5M_00001_WR100%/APnL120%/PTPnL2%

  • {TICKER} — Full ticker symbol (AMZN, CLSK, AAL). No abbreviations.
  • {TYPE} — S=standalone, L=lead-lag, D=divergence, X=inverse
  • {TF} — 5M, 15M, 1H, DAILY (always UPPERCASE)
  • {SEQ5} — 5-digit zero-padded sequence. Bull rules: 00001-04999. Bear rules: 05001+
  • _WR.../APnL.../PTPnL... — Backtest metrics suffix embedded in the rule ID

Family keys in ALL_FAMILIES: {TICKER}_S, {TICKER}_L, {TICKER}_D, {TICKER}_X, {TICKER}_B1_S (bear), etc.

2.3 Bull/Bear Number Separation NEW in v7

Bear Rule Numbering at 5001+

Bull rules use sequence numbers 00001–04999. Bear rules start at 05001+. This prevents base rule ID collisions where a bull CLSK_S5M_00003 and a bear CLSK_S5M_00003 shared the same base ID, causing the wrong stop-loss parameters to be used.

  • Implemented in create_rules_local3.py: BEAR_RULE_START = 5001
  • Migration script: renumber_bear_rules.py shifts existing bear rules to 05001+ range
  • Affects: index blocks, rule files, rule_index.py, backtest_all_results.json

2.4 Column Definitions

Column Type Description
Rule_ID String Unique rule identifier. Format: {TICKER}_{TYPE}{TF}_{SEQ5}_WR/APnL/PTPnL. The base ID (before _WR) is the lookup key across paper traders and rule files.
Ticker String The stock/ETF ticker the rule trades (e.g., AMZN, AAPL, CLSK).
Timeframe String Native evaluation timeframe: 5M, 15M, 1H, or DAILY.
Tier Integer Backtest quality tier (1 = best, 2 = strong). Only T1 and T2 rules are in production. See Section 5.3 for criteria.
Regime NEW String Allowed market regime: BULL or BEAR. Regime gate prevents rules from firing in the wrong market condition.
Win_Rate Percent Backtest win rate from creation pipeline.
Avg_PnL Percent Backtest average PnL per trade.
Correlations String Lead / Same / Inverse tickers used during rule creation (e.g., MSFT / QQQ / SQQQ).

2.5 Design Notes

  • No Status column — everything on the list is active. Inactive/retired rules are removed.
  • CHANGED from v6 Last_Run_Time column removed — not implemented in production; adds tracking overhead without clear value.
  • Rule index is rule_index.py (Python module with ALL_FAMILIES dict), not rule_index.json.
  • Table should be searchable (by ticker, rule ID) and sortable by any column.
  • Mobile-friendly: horizontal scroll on small screens, dark theme.

3. Unified Trader Architecture

3.1 Core Principle: Individual = Master, Stock Pool = Follower

Architecture Summary

The individual trader (paper_trader_individual.py) is the primary decision maker. It evaluates rules, decides to buy or sell, and executes trades on its own portfolio. The stock pool trader (paper_trader_stock.py) mirrors the individual trader's actions when possible. The crypto traders operate independently.

3.2 Regime Gating NEW in v7

Market Regime Filter

Each rule has a regime_allowed field (BULL or BEAR). The paper trader maintains a current regime state and gates rule evaluation accordingly.

  • REGIME_ENABLED = True in paper_trader_individual.py
  • Regime gate runs inside the evaluation loop (not at position-open time) to prevent signal shadowing
  • If a bear rule fires first in the eval loop, the regime gate skips it via continue, allowing the next valid bull rule to fire
  • Safety net: a second regime check remains in _open_position as a defense-in-depth layer

3.3 Rule Dedup NEW in v7

Python Object Identity Dedup

Rule lists are deduped using id(rule_list) rather than prefix string matching. This correctly handles:

  • Case aliases: ALL_AZS_5m = ALL_AZS_5M (same object, loaded once)
  • Legacy aliases: ALL_LTS = ALL_LMTS (same object, loaded once)
  • Distinct families: CLSK_B2_S != CLSK_S (different objects, both loaded)

3.4 Trade Flow: Buy Signal

Individual Trader evaluates rules | v Regime gate: is rule allowed in current regime? | / \ / \ Allowed Not allowed | | v v Continue Skip (continue to next rule) evaluation | v Rule triggers BUY for ticker X | v Individual Trader executes BUY (own portfolio, own balance) | v Signal sent to Stock Pool Trader | v Stock Pool checks available balance | / \ / \ Balance No balance available available | | v v Execute Skip trade BUY (log: "no balance")

3.5 Trade Flow: Sell Signal

Individual Trader evaluates rules | v Rule triggers SELL for ticker X | v Individual Trader executes SELL (closes own position) | v Signal sent to Stock Pool Trader | v Stock Pool checks if ticker X has an open position | / \ / \ Position No position exists for ticker X | | v v Execute Skip SELL (no action needed)

3.6 Key Rules

  1. Individual trader always trades. Its decisions are independent of the stock pool's state.
  2. Stock pool never initiates. It only acts in response to signals from the individual trader.
  3. Regime gating at eval time. Rules blocked by regime are skipped via continue so the next valid rule can fire.
  4. Balance gating on buy. Stock pool only mirrors a buy if it has sufficient available balance.
  5. Position check on sell. Stock pool only mirrors a sell if it currently holds that ticker.
  6. No partial fills. Stock pool either mirrors the full trade or skips entirely.
  7. Logging. Every mirror attempt (success or skip) is logged with the reason.

3.7 Implementation Details

  • Crypto traders operate independently with their own rules, balance, and 24/7 Binance schedule:
    • paper_trader_crypto.py — per-rule BTC trader over the production BTC rule set (~274 rules).
    • paper_trader_crypto_v2.py NEW — consensus trader over the same rule set: enters on ≥2 concurrent signals, hold-override at 50% agreement, exit-flip at 30%. Reports net consensus (e.g. 1B/0S of 274) per cycle. A v1-vs-v2 comparison report is generated in backtest_reports/.
  • Signal mechanism: individual trader writes trade events for pool mirroring.
  • Mirror happens within the same evaluation cycle, not on a delay.

3.8 GPU ML Veto Filter NEW in v8 DORMANT

PatternLSTM signal filter — present but inactive

The individual trader loads an optional GPU model (ml/gpu_predictor.py → GPUSignalFilter, weights at models/gpu/pattern_lstm.pt) intended to veto rule signals: when a rule fires a signal, the LSTM can block it if it predicts the opposite direction. It never generates trades on its own.

  • Status: dormant / not trusted. Zero GPU VETO events across the entire trade log; the model validated at 20.6% accuracy (3-class, below random) and was trained on 50 tickers unrelated to the traded universe; it CUDA-crashed on 2026-06-26.
  • An offline skill test (2026-07-21) confirmed the model's directional calls are ≤ coin-flip on traded tickers, so activating it would strip winners. It remains wired in but effectively no-ops.
  • Superseded by the meta-label approach — see Section 13.2. Any future signal-quality gating should come from that, not this model.

5. Rule Creation & Promotion Pipeline

5.1 Pipeline Overview

The create-rules pipeline uses a 13-phase architecture with two optimization loops. Rules are never filtered during optimization — a ratchet mechanism ensures rules can only improve or stay the same. Tier labeling after optimization is the only filter.

Pipeline File Optimization Use Case
create-rules3 create_rules_local3.py 13-phase: 5-cycle backtest loop + sim optimization loop Standard rule creation for all tickers. Runs locally with ProcessPoolExecutor (up to 8 workers). Zero API tokens consumed.
create-rules-bear NEW create_rules_bear.py Bear-specific rule generation Chained after bull pipeline via --chain-bear. Bear rules start at sequence 05001+.
REMOVED create_rules_local5.py — referenced in v6 but never existed. create_rules_local3.py is the 5-cycle pipeline and is the only production pipeline.

5.4 Pipeline Phases

Pre-Pipeline: Setup

Phase 0: Correlation Research Auto-search broad ticker universe Find lead, same-direction, and inverse tickers | v Data Download & Preparation Polygon.io for ALL stock timeframes (5m, 15m, 1h, daily) Binance for crypto Retry with exponential backoff on connection errors

Backtest Optimization Loop (all rules, no filtering)

Phase 1: Generate all parameter combinations and backtest each one Indicator combos x TP/SL x hold periods x directions No filtering -- ALL rules survive | v Phase 2: Optimize all rules by tweaking parameters Adjust TP/SL ratios, hold periods, indicator thresholds Goal: improve WR and PnL for every rule | v Phase 3: Re-backtest all rules with new parameters | v Phase 4: Per-rule ratchet comparison If rule improved (WR or PnL) -> keep new parameters If rule got worse -> revert to prior parameters Rules can only improve or stay the same | v Phase 5: Repeat Phases 2-4 (5 cycles total) Each cycle: optimize -> backtest -> compare -> ratchet

Tier Gate (the ONLY filter)

Phase 6: Apply Tier 1 & Tier 2 labels Dual-path T1 (100%WR/10+ trades OR 90%WR/15+ trades) T2: 70%WR + 10%APnL + 5+ trades Unlabeled rules are discarded This is the ONLY point where rules are filtered out

Simulation Optimization Loop (Tier 1/2 only)

Phase 7: Run 1-minute simulation on Tier 1/2 rules (baseline) Replays every rule on 1m bars for intrabar TP/SL accuracy | v Phase 8: Optimize Tier 1/2 rules by tweaking parameters | v Phase 9: Re-run 1-minute simulation with new parameters | v Phase 10: Per-rule ratchet comparison Report % delta for WR and PnL from Phase 7 baseline | v Phase 11: Pause for human approval (temporary -- commentable) | v Phase 12: If approved and optimization still improving, repeat 8-11 | v Phase 13: Deploy pipeline Rule files + index blocks + reports + dashboards ONLY HQ runs Phase 13 (workers use --skip-deploy)

5.5 Rule Promotion Failure Points

Failure Mode 1: Import Validation Errors

New rule files may reference indicator functions the paper trader doesn't recognize. Crashes on startup with NameError or ImportError.

  • Prevention: Compile-only validation (no trader startup) during pipeline.
Failure Mode 2: Timeframe Case Mismatch

Rule exports must use UPPERCASE timeframes (5M, 15M, 1H, DAILY). Lowercase exports cause silent import failures.

  • Prevention: Pipeline enforces uppercase on export. Recurring issue — validated every time.
Failure Mode 3: Bull/Bear ID Collision NEW

Bull and bear rules sharing the same base sequence number (e.g., both CLSK_S5M_00003) caused the wrong stop-loss parameters to be used. A bear rule's 2.5% SL overwrote a bull rule's 14.88% SL.

  • Prevention: Bear rules start at 05001+. renumber_bear_rules.py migration script.
  • Additional defense: Regime gate blocks out-of-regime rules; id()-based dedup prevents object aliasing.
Failure Mode 4: Trader Process Not Restarted

New rules deployed but running trader still has old rules in memory.

  • Current approach: Pipeline does NOT restart traders automatically. Uses compile-only validation.
  • Required: Hot-reload mechanism or safe restart window.

5.6 Pipeline Safeguards (Current)

  • Max 1 concurrent pipeline per PC — strictly serial, no parallel ever
  • Compile-only import validation (no trader startup during pipeline)
  • Uppercase timeframe enforcement at all boundaries
  • JS syntax validation on index.html before GitHub Pages push
  • Backtest report generated with unique timestamped filename (never overwrite)
  • Rules always run on their native timeframe indicators
  • Ratchet-only optimization — rules can never get worse between cycles
  • Retry with exponential backoff on Polygon API connection errors
  • Distributed Polygon API lock across 3-PC cluster
  • Bear rules numbered at 05001+ to prevent ID collision with bull rules

5.7 Data Pipeline Enhancements NEW in v8

Data hygiene & enrichment (June 2026)
  • Timezone normalization: all historic intraday CSVs normalized to US/Pacific. New Polygon downloads still require re-stamping into US/Pacific before use.
  • Daily event flags: daily CSVs carry split / dividend / earnings event columns (annotated via annotate_daily_events.py). Note: the pipeline currently overwrites these on re-download — re-annotation is an open integration TODO.
  • Correlation universe: quality-gated to ~2,653 tickers (built by build_correlation_universe.py), replacing the old hardcoded list. Correlation research searches this broad universe, avoiding thin-data tickers.
  • Source policy (confirmed): Polygon for stocks, Binance for crypto.

5.8 Rule Metrics Ledger & Nightly Reconcile NEW in v8

Live-trade ledger & self-healing inventory
  • rule_metrics_ledger.py records every closed live paper trade into rule_metrics.db (SQLite/WAL): rule_id, ticker, entry/exit price, pnl, win, exit_reason, opened/closed timestamps, bars_held.
  • Feeds trade-alert stats and combined per-rule metrics; the ledger is also the training source for the meta-label model (Section 13.2).
  • Nightly reconcile self-heals rule_inventory drift — production rules that were missing from the ledger get registered automatically, keeping tier/status in sync.

6. 2-Way Telegram Communication

6.1 Architecture

The system uses a bridge process to communicate between Claude Code and Steven's Telegram. Only one component may call the Telegram API — the bridge. This prevents 409 Conflict errors from multiple consumers.

Steven's Telegram App | v Telegram Bot API (getUpdates, long-poll) | v telegram_bridge.py (PID file, single instance) | | v v telegram_inbox.json telegram_outbox.json (bridge WRITES) (bridge READS & sends) | ^ v | Claude Code watch loop (READS inbox, WRITES outbox)

6.2 Message Structure

Inbox (telegram_inbox.json)

FieldTypeDescription
messageStringMessage text content from Steven
senderStringTelegram sender name
reply_chatIntegerTelegram chat ID for reply routing
timestampISO 8601When the message was received by the bridge
statusString"pending" or "processed"
processed_atISO 8601When the watch loop processed the message

Outbox (telegram_outbox.json)

FieldTypeDescription
messageStringResponse text to send to Steven
timestampISO 8601When the response was queued
statusString"pending" → bridge sends and removes entry

6.3 Bridge Process (telegram_bridge.py)

  • Runs as a detached background process (subprocess.Popen with DETACHED_PROCESS)
  • Single-instance enforcement: kills existing bridge PIDs before starting
  • Polls Telegram API every 3 seconds with 25s long-poll timeout
  • Uses requests library for reliable Windows timeout handling
  • Threading hard-deadline: 90s timeout on getUpdates to prevent silent hangs
  • Heartbeat written to telegram_bridge_heartbeat.json every poll cycle
  • Auto-respawn after 10 consecutive errors

6.4 Watch Loop (Claude Code)

  • Polls telegram_inbox.json using Python json.load(encoding='utf-8')
  • Controlled by telegram_watch.json: {"active": true/false, "last_check_at": ..., "empty_checks": N}
  • Wake-up: check active flag first, read inbox, process pending, write outbox, schedule next
  • P0 rule: NEVER call Telegram API directly. Only bridge calls getUpdates.
  • Adaptive polling: 270s when active, widening to 1200s after consecutive empty checks

13. ML & Analytics Layer NEW in v8

A research/ranking/ML layer built June–July 2026. Everything here is analysis and monitoring — none of it currently places or gates live trades. Trade decisions still come entirely from the deterministic rule engine (Section 3).

13.1 Ticker Ranking Engine (probability_engine.py)

5-factor forward-looking rank of the deployed-rule universe (~331 tickers)

Composite score = EV × regime × arb-edge × strength × decay, where EV = target% × P(hit).

  • Future price — target blends each ticker's rule-backtested PTPnL% with a vol-scaled 1σ move; P(hit) via Black-Scholes d2 on one shared volatility estimate (cached ATM IV blended with realized vol).
  • Company strength — DOMINANT lever: profit margin, ROE, revenue & earnings growth, debt/equity, free cash flow (yfinance) mapped to a factor in [0.25, 1.0] (STRENGTH_WEIGHT=0.75). ETFs pass through at 1.0. Cached to data/fundamentals_cache.csv (14-day refresh).
  • Regime — TickerRegimeFilter state (BULL/CAUTION/BEAR) from regime_suite.
  • ETF decay — leveraged/inverse variance drag exp(−½·L(L−1)·σ²·T), applied at each fund's exact per-ETF value (DECAY_WEIGHT=1.0). Plain equities L=1 → 1.0. Leverage L parsed from yfinance fund name.
  • Arb edge — cached options-mispricing score where available, else a realized-vol-compression heuristic.

Output: logs/probability_engine_rank.csv and the hub page ticker_rankings.html (weekly RefreshTickerRankings). Design intent: junk daily-reset ETFs stay demoted by decay while company fundamentals drive the equity ordering.

13.2 Meta-Label Model & Shadow Forward Test

"Will THIS rule signal win?" — offline-validated, forward-testing in shadow

A secondary model over the trade ledger (Section 5.8) that scores each fired rule signal's probability of winning — the classic meta-labeling pattern. It never invents trades; it would only decline or size down low-probability signals.

  • Data: 1,190 real paper trades (55.9% base WR). Ledger stores outcomes only; entry-state features are reconstructed from native-timeframe bars at entry (respecting the native-timeframe rule).
  • Features: rule priors (backtest WR/PTPnL/tier/confidence) + entry market state (RSI, stochastic, momentum, realized vol, ATR, distance-from-SMA, volume, regime proxy) + direction-signed momentum.
  • Validation: purged walk-forward (train on trades closed before the test window minus embargo). Out-of-sample AUC 0.581 (logistic baseline 0.532). Threshold sweep is cleanly monotonic; at P(win)≥0.60 win rate rises to ~62.7% (vs 54.5%) keeping ~41% of trades. ~58% of predictive gain comes from market-state features, not just rule quality.
  • Files: ml_meta_label_eval.py (offline eval + report), meta_label_train.py (freezes model to models/meta_label/), meta_label_shadow.py (scorer).
  • Shadow forward test (live, log-only): the frozen model scores every trade opened after its training cutoff from entry-time features, logging P(win) + outcome to a separate meta_shadow.db and the hub page meta_label_shadow.html. Fully decoupled — zero paper-trader edits, no live action. Hourly MetaLabelShadow task.
Deployment gate: only after the shadow test accrues enough unseen trades (~1–2 months) and holds up would this be wired in — and then as confidence-weighted sizing, not a hard veto, inside a controlled restart window (Section 15.2). Nothing touches trading until then.
