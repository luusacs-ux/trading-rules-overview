1. Document Purpose

This document defines the design requirements for the next iteration of the trading system. It covers three areas:

  1. Unified Rules Tab — a single view of all active rules across all tickers and timeframes
  2. Unified Trader Architecture — how the individual trader and stock pool trader interact

Status: Requirements only — no code changes yet. This document is for review and approval before implementation begins.

2. Unified Rules Tab

2.1 Purpose

A single, searchable table showing every active trading rule. Replaces the need to look up rules per-ticker. Every rule on this list is active — there is no status column.

2.2 Column Definitions

Column Type Description
Rule_ID String Unique rule identifier. Format: {PREFIX}{TF_CODE}_{SEQ} (e.g., AMZNS_001). The prefix encodes the ticker, the TF code encodes the timeframe family (S=5m, L=15m, D=1h, X=daily), and the sequence number is unique within that family. This is the same ID used in rule_index.json and paper traders.
Ticker String The stock/ETF ticker the rule trades (e.g., AMZN, AAPL, BTC).
Timeframe String Native evaluation timeframe: 5M, 15M, 1H, or DAILY.
Tier Integer Backtest quality tier (1 = best, 3 = baseline). Derived from win rate and PnL during rule creation.
Win_Rate Percent Backtest win rate from creation pipeline.
Avg_PnL Percent Backtest average PnL per trade.
Last_Run_Time Datetime The last time this rule was evaluated by the paper trader. Records when the rule was considered for a trade or rejected for a trade. Updated on every evaluation cycle, not just on trade execution.
Correlations String Lead / Same / Inverse tickers used during rule creation (e.g., MSFT / QQQ / SQQQ).

2.3 Design Notes

  • CHANGE No Status column — everything on the list is active. Inactive/retired rules are simply removed from the list.
  • NEW Last_Run_Time column tracks when the paper trader last evaluated the rule, whether it triggered a trade or not.
  • Rule ID must match the format used by rule_index.json so lookups are consistent across the dashboard, paper traders, and rule files.
  • Table should be searchable (by ticker, rule ID) and sortable by any column.
  • Mobile-friendly: horizontal scroll on small screens, same dark theme as rules overview page.

3. Unified Trader Architecture

3.1 Core Principle: Individual = Master, Stock Pool = Follower

Architecture Summary

The individual trader is the primary decision maker. It evaluates rules, decides to buy or sell, and executes trades on its own portfolio. The stock pool trader is a follower — it does not make independent trading decisions. It mirrors the individual trader's actions when possible.

3.2 Trade Flow: Buy Signal

Individual Trader evaluates rules | v Rule triggers BUY for ticker X | v Individual Trader executes BUY (own portfolio, own balance) | v Signal sent to Stock Pool Trader | v Stock Pool checks available balance | / \ / \ Balance No balance available available | | v v Execute Skip trade BUY (log: "no balance")

3.3 Trade Flow: Sell Signal

Individual Trader evaluates rules | v Rule triggers SELL for ticker X | v Individual Trader executes SELL (closes own position) | v Signal sent to Stock Pool Trader | v Stock Pool checks if ticker X has an open position | / \ / \ Position No position exists for ticker X | | v v Execute Skip SELL (no action needed)

3.4 Key Rules

  1. Individual trader always trades. It never checks the stock pool's state before acting. Its decisions are independent.
  2. Stock pool never initiates. It only acts in response to signals from the individual trader. It does not evaluate rules on its own.
  3. Balance gating on buy. Stock pool only mirrors a buy if it has sufficient available balance. If not, the trade is skipped (not queued).
  4. Position check on sell. Stock pool only mirrors a sell if it currently holds that ticker. If not, no action is taken.
  5. No partial fills. Stock pool either mirrors the full trade or skips entirely.
  6. Logging. Every mirror attempt (success or skip) is logged with the reason.

3.5 Implementation Considerations

  • Signal mechanism: individual trader writes trade events to a shared signal file or calls a function directly. Exact mechanism TBD during implementation.
  • Timing: mirror should happen within the same evaluation cycle, not on a delay.
  • The crypto trader operates independently — it is not part of this master/follower architecture. Crypto has its own rules, its own balance, and its own 24/7 schedule.
