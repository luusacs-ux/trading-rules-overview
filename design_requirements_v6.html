2. Unified Rules Tab

2.1 Purpose

A single, searchable table showing every active trading rule. Replaces the need to look up rules per-ticker. Every rule on this list is active — there is no status column.

2.2 Column Definitions

Column Type Description
Rule_ID String Unique rule identifier. Format: {PREFIX}{TF_CODE}_{SEQ} (e.g., AMZNS_001). The prefix encodes the ticker, the TF code encodes the timeframe family (S=5m, L=15m, D=1h, X=daily), and the sequence number is unique within that family. This is the same ID used in rule_index.json and paper traders.
Ticker String The stock/ETF ticker the rule trades (e.g., AMZN, AAPL, BTC).
Timeframe String Native evaluation timeframe: 5M, 15M, 1H, or DAILY.
Tier Integer Backtest quality tier (1 = best, 3 = baseline). Derived from win rate and PnL during rule creation.
Win_Rate Percent Backtest win rate from creation pipeline.
Avg_PnL Percent Backtest average PnL per trade.
Last_Run_Time Datetime The last time this rule was evaluated by the paper trader. Records when the rule was considered for a trade or rejected for a trade. Updated on every evaluation cycle, not just on trade execution.
Correlations String Lead / Same / Inverse tickers used during rule creation (e.g., MSFT / QQQ / SQQQ).

2.3 Design Notes

  • CHANGE No Status column — everything on the list is active. Inactive/retired rules are simply removed from the list.
  • NEW Last_Run_Time column tracks when the paper trader last evaluated the rule, whether it triggered a trade or not.
  • Rule ID must match the format used by rule_index.json so lookups are consistent across the dashboard, paper traders, and rule files.
  • Table should be searchable (by ticker, rule ID) and sortable by any column.
  • Mobile-friendly: horizontal scroll on small screens, same dark theme as rules overview page.

3. Unified Trader Architecture

3.1 Core Principle: Individual = Master, Stock Pool = Follower

Architecture Summary

The individual trader is the primary decision maker. It evaluates rules, decides to buy or sell, and executes trades on its own portfolio. The stock pool trader is a follower — it does not make independent trading decisions. It mirrors the individual trader's actions when possible.

3.2 Trade Flow: Buy Signal

Individual Trader evaluates rules | v Rule triggers BUY for ticker X | v Individual Trader executes BUY (own portfolio, own balance) | v Signal sent to Stock Pool Trader | v Stock Pool checks available balance | / \ / \ Balance No balance available available | | v v Execute Skip trade BUY (log: "no balance")

3.3 Trade Flow: Sell Signal

Individual Trader evaluates rules | v Rule triggers SELL for ticker X | v Individual Trader executes SELL (closes own position) | v Signal sent to Stock Pool Trader | v Stock Pool checks if ticker X has an open position | / \ / \ Position No position exists for ticker X | | v v Execute Skip SELL (no action needed)

3.4 Key Rules

  1. Individual trader always trades. It never checks the stock pool's state before acting. Its decisions are independent.
  2. Stock pool never initiates. It only acts in response to signals from the individual trader. It does not evaluate rules on its own.
  3. Balance gating on buy. Stock pool only mirrors a buy if it has sufficient available balance. If not, the trade is skipped (not queued).
  4. Position check on sell. Stock pool only mirrors a sell if it currently holds that ticker. If not, no action is taken.
  5. No partial fills. Stock pool either mirrors the full trade or skips entirely.
  6. Logging. Every mirror attempt (success or skip) is logged with the reason.

3.5 Implementation Considerations

  • Signal mechanism: individual trader writes trade events to a shared signal file or calls a function directly. Exact mechanism TBD during implementation.
  • Timing: mirror should happen within the same evaluation cycle, not on a delay.
  • The crypto trader operates independently — it is not part of this master/follower architecture. Crypto has its own rules, its own balance, and its own 24/7 schedule.

5. Rule Creation & Promotion Pipeline

Known issue: Rule promotion (deploying new rules into the paper traders) tends to break the trader processes. This section documents the full pipeline and identifies the failure points that must be addressed.

5.1 Pipeline Overview

Two pipeline variants exist for creating trading rules. Both follow the same multi-phase process but differ in optimization depth:

Pipeline File Optimization Use Case
create-rules3 create_rules_local3.py 5-cycle adaptive grid optimization Standard rule creation — used for most tickers. Runs locally with ProcessPoolExecutor (up to 8 workers). Zero API tokens consumed.
create-rules5 create_rules_local5.py Extended optimization with additional validation passes Deep optimization for high-priority or complex tickers. Longer runtime but higher quality rule output.

5.2 Pipeline Phases

Phase 0: Correlation Research Find lead, same-direction, and inverse tickers (auto-research broad universe or manual override) | v Phase 1: Data Download & Preparation Download/update historical data for ticker + correlations Timeframes: 5m, 15m, 1h, daily | v Phase 2: Rule Generation (per timeframe) Generate candidate rules using indicator combinations Backtest each candidate against historical data Multi-cycle optimization: refine parameters each cycle | v Phase 3: Rule Selection & Tiering Rank rules by win rate + PnL Assign tiers: Tier 1 (best), Tier 2, Tier 3 Generate rule files with prefix naming | v Phase 4: Deployment <-- BREAKAGE POINT a) Update rule_index.json b) Generate backtest report (HTML) c) Deploy to paper traders (import validation) d) Update dashboard / GitHub Pages

5.3 Rule Promotion Failure Points

Phase 4 deployment is where the paper traders tend to break. Known failure modes:

Failure Mode 1: Import Validation Errors

New rule files may reference indicator functions or parameters that the paper trader doesn't recognize. The trader crashes on startup with NameError or ImportError.

  • Example: NameError: name 'fetch_coin_5m' is not defined — rule referenced wrong function name
  • Prevention: Compile-only validation (no trader startup) during pipeline. Verify all indicator function names exist before deploying.
Failure Mode 2: Timeframe Case Mismatch

Rule exports must use UPPERCASE timeframes (5M, 15M, 1H, DAILY). Lowercase exports (5m) cause silent import failures — rules are loaded but never matched to incoming data.

  • Prevention: Pipeline enforces uppercase on export. Recurring issue — must be validated every time.
Failure Mode 3: Rule Index Corruption

Concurrent pipeline runs writing to rule_index.json can corrupt the file (partial writes, missing commas, duplicate entries). The paper trader fails to parse the index and crashes.

  • Prevention: Max 2 concurrent pipelines. File locking on rule_index.json writes. JSON syntax validation after every write.
Failure Mode 4: Trader Process Not Restarted

New rules are deployed to rule files and index, but the running paper trader still has the old rules in memory. The trader must be restarted to pick up new rules, but restarting during market hours risks missing trades on existing rules.

  • Current approach: Pipeline does NOT restart traders automatically. Uses compile-only validation to avoid sending startup notifications during pipeline runs.
  • Required: Hot-reload mechanism or safe restart window.

5.4 Pipeline Safeguards (Current)

  • Max 2 concurrent pipeline runs (prevents index corruption)
  • Compile-only import validation (no trader startup during pipeline)
  • Uppercase timeframe enforcement on all exports
  • JSON syntax validation on index.html before GitHub Pages push
  • Backtest report generated with unique timestamped filename (never overwrite)
  • Rules always run on their native timeframe indicators

5.5 Required Improvements

  1. Pre-deployment smoke test: Before adding rules to rule_index.json, run a lightweight import test that verifies every function referenced in the rule exists in the trader's namespace.
  2. Atomic index update: Write to a temp file, validate JSON, then rename — prevents partial writes from corrupting rule_index.json.
  3. Hot-reload support: Paper traders should be able to pick up new rules without a full restart. Watch rule_index.json for changes and reload on modification.
  4. Rollback on failure: If a new rule set causes the trader to crash, automatically revert to the previous rule_index.json and alert via Telegram.
  5. Post-deploy health check: After rule promotion, verify the paper trader is still running and processing rules within 60 seconds. Alert if not.

6. 2-Way Telegram Communication

6.1 Architecture

The system uses a bridge process to communicate between Claude Code and Steven's Telegram. Only one component may call the Telegram API — the bridge. This prevents 409 Conflict errors from multiple consumers.

Steven's Telegram App | v Telegram Bot API (getUpdates, long-poll) | v telegram_bridge.py (PID file, single instance) | | v v telegram_inbox.json telegram_outbox.json (bridge WRITES) (bridge READS & sends) | ^ v | Claude Code watch loop (READS inbox, WRITES outbox)

6.2 Message Structure

Inbox (telegram_inbox.json)

Array of message objects. Bridge appends new messages; Claude Code reads and marks as processed.

FieldTypeDescription
messageStringMessage text content from Steven
senderStringTelegram sender name
reply_chatIntegerTelegram chat ID for reply routing
timestampISO 8601When the message was received by the bridge
statusString"pending" (unread) or "processed" (handled by watch loop)
processed_atISO 8601When the watch loop processed the message (added on processing)

Outbox (telegram_outbox.json)

Array of response objects. Claude Code writes; bridge reads, sends via Telegram API, and clears.

FieldTypeDescription
messageStringResponse text to send to Steven
timestampISO 8601When the response was queued
statusString"pending" (unsent) → bridge sends and removes entry

6.3 Bridge Process (telegram_bridge.py)

  • Runs as a detached background process (subprocess.Popen with DETACHED_PROCESS)
  • Single-instance enforcement: kills existing bridge PIDs before starting
  • Polls Telegram API every 3 seconds with 25s long-poll timeout
  • Uses requests library (not urllib) for reliable Windows timeout handling
  • Threading hard-deadline: 90s timeout on getUpdates to prevent silent hangs
  • Heartbeat written to telegram_bridge_heartbeat.json every poll cycle
  • Auto-respawn after 10 consecutive errors
  • Crash log: logs/telegram_bridge_crash.log

6.4 Watch Loop (Claude Code)

  • Polls telegram_inbox.json using Python json.load(encoding='utf-8') — never ripgrep (encoding failures on non-ASCII)
  • Controlled by telegram_watch.json: {"active": true/false, "last_check_at": ..., "empty_checks": N}
  • Wake-up procedure: (1) check active flag, (2) read inbox, (3) process pending, (4) write outbox, (5) schedule next wake-up
  • P0 rule: NEVER call Telegram API directly. Only bridge calls getUpdates.
  • Adaptive polling: 270s when active, widening to 1200s after consecutive empty checks

6.5 Known Issues & Mitigations

#IssueRoot CauseMitigation
1Bridge hangs silentlyurllib SSL timeout unreliable on WindowsReplaced with requests + 90s thread deadline

8. Regression Test Suite

8.1 Purpose

Automated validation that catches data issues, deployment mismatches, and dashboard corruption before they affect paper trading. Runs on-demand or after pipeline deployments.

8.2 Test Steps

StepNameWhat It ChecksPass Criteria
1 Data Completeness All tickers in rule_index have historical data files for all 4 timeframes (5m, 15m, 1h, daily). Checks for ≥85% of expected trading days. ≥85% completeness per timeframe per ticker
2 Data Accuracy Cross-timeframe alignment: aggregates 5m/15m bars into 1h/daily OHLC and compares. Catches source mismatches (yfinance adjusted prices vs raw). <1% mean price deviation between aggregated and native timeframe
3 Rule Deployment All tickers in rule_index.json are deployed to both paper traders (individual + stock pool). Rule files exist and are importable. Every indexed rule has a matching deployed file; compile check passes
4 Dashboard Integrity All tickers appear in gh-pages index.html. Report links resolve to existing HTML files. JavaScript data array has valid syntax. All tickers present, all links valid, JS parses without error

8.3 Invocation

MethodCommand
CLIpython regression_test.py
CLI with auto-fixpython regression_test.py --fix
Specific steps onlypython regression_test.py --step 1,3
Telegram/regression or /regtest
Dashboard buttonBoth dashboard_v4.pyw and dashboard_individual.pyw have a regression test button

8.4 Auto-Fix Behavior (--fix flag)

  • Step 1: Re-downloads missing data files using appropriate source (yfinance/Alpha Vantage/Twelve Data)
  • Step 2: Re-downloads from single source if cross-source mismatch detected
  • Step 3: Re-runs import validation; cannot auto-fix rule code errors (requires manual review)
  • Step 4: Regenerates index.html from current rule data; fixes missing report links

8.5 Required Improvements

  1. Post-pipeline auto-run: Automatically run regression test after every create-rules pipeline completes
  2. Telegram results: Send regression test pass/fail summary to Steven via Telegram after completion
  3. Historical tracking: Log regression results with timestamps so trends in data quality or deployment issues are visible over time
